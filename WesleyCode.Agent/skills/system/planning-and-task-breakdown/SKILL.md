# 规划与任务拆解

## 概览

把工作拆成小型、可验证、带明确验收标准的任务。好的任务拆解能让智能体稳定完成工作，而不是产出纠缠不清的实现。每个任务都应足够小，可以在一次专注会话中实现、测试和验证。

## 何时使用

- 已有规格说明，需要拆成可实现单元
- 任务太大或太模糊，难以直接开始
- 工作需要分配给多个智能体或多个会话并行执行
- 需要向人类清楚沟通范围
- 实现顺序不明显

**不应使用：** 范围明显的单文件修改，或规格说明已经包含定义良好的任务。

## 规划流程

### 第 1 步：进入规划模式

写任何代码之前，先以只读方式工作：

- 阅读规格说明和相关代码区域
- 识别现有模式和约定
- 映射组件之间的依赖关系
- 记录风险和未知项

**规划期间不要写代码。** 输出应是计划文档，而不是实现。

### 第 2 步：识别依赖图

梳理谁依赖谁：

```text
数据库结构
  -> API 模型/类型
      -> API 端点
          -> 前端 API 客户端
              -> UI 组件
      -> 验证逻辑
  -> 种子数据/迁移
```

实现顺序应沿依赖图自底向上推进：先构建基础，再构建上层能力。

### 第 3 步：纵向切片

不要先做完所有数据库、再做所有 API、最后做所有 UI。应一次构建一条完整的功能路径。

**不推荐（横向切片）：**

```text
任务 1：构建完整数据库结构
任务 2：构建所有 API 端点
任务 3：构建所有 UI 组件
任务 4：连接所有内容
```

**推荐（纵向切片）：**

```text
任务 1：用户可以创建账号（注册所需的结构 + API + UI）
任务 2：用户可以登录（认证结构 + API + 登录 UI）
任务 3：用户可以创建任务（任务结构 + API + 创建 UI）
任务 4：用户可以查看任务列表（查询 + API + 列表 UI）
```

每个纵向切片都应交付可运行、可测试的功能。

### 第 4 步：编写任务

每个任务使用以下结构：

```markdown
## Task [N]: [简短描述性标题]

**Description:** 用一段话说明此任务要完成什么。

**Acceptance criteria:**
- [ ] [具体、可测试的条件]
- [ ] [具体、可测试的条件]

**Verification:**
- [ ] Tests pass: `npm test -- --grep "feature-name"`
- [ ] Build succeeds: `npm run build`
- [ ] Manual check: [要人工验证的内容]

**Dependencies:** [依赖的任务编号，或 "None"]

**Files likely touched:**
- `src/path/to/file.ts`
- `tests/path/to/test.ts`

**Estimated scope:** [Small: 1-2 files | Medium: 3-5 files | Large: 5+ files]
```

### 第 5 步：排序并设置检查点

安排任务时确保：

1. 依赖已满足，基础能力先完成
2. 每个任务完成后系统仍处于可运行状态
3. 每 2-3 个任务后设置验证检查点
4. 高风险任务靠前，尽早暴露问题

加入明确检查点：

```markdown
## Checkpoint: After Tasks 1-3
- [ ] All tests pass
- [ ] Application builds without errors
- [ ] Core user flow works end-to-end
- [ ] Review with human before proceeding
```

## 任务大小指南

| 大小 | 文件数 | 范围 | 示例 |
|------|--------|------|------|
| **XS** | 1 | 单个函数或配置修改 | 添加一条验证规则 |
| **S** | 1-2 | 一个组件或端点 | 添加新的 API 端点 |
| **M** | 3-5 | 一个功能切片 | 用户注册流程 |
| **L** | 5-8 | 多组件功能 | 带过滤和分页的搜索 |
| **XL** | 8+ | **过大，需要继续拆分** | 完整计费系统 |

如果任务达到 L 或更大，应继续拆成更小任务。智能体最适合执行 S 和 M 任务。

**需要继续拆分的信号：**

- 需要超过一次专注会话，约 2 小时以上智能体工作量
- 无法用 3 条以内验收标准描述清楚
- 同时触及两个或更多独立子系统，例如认证和计费
- 任务标题里出现“并且”，通常说明它其实是两个任务

## 计划文档模板

```markdown
# Implementation Plan: [Feature/Project Name]

## Overview
[用一段话概述要构建的内容]

## Architecture Decisions
- [关键决策 1 及理由]
- [关键决策 2 及理由]

## Task List

### Phase 1: Foundation
- [ ] Task 1: ...
- [ ] Task 2: ...

### Checkpoint: Foundation
- [ ] Tests pass, builds clean

### Phase 2: Core Features
- [ ] Task 3: ...
- [ ] Task 4: ...

### Checkpoint: Core Features
- [ ] End-to-end flow works

### Phase 3: Polish
- [ ] Task 5: ...
- [ ] Task 6: ...

### Checkpoint: Complete
- [ ] All acceptance criteria met
- [ ] Ready for review

## Risks and Mitigations
| Risk | Impact | Mitigation |
|------|--------|------------|
| [Risk] | [High/Med/Low] | [Strategy] |

## Open Questions
- [Question needing human input]
```

## 并行化机会

当有多个智能体或会话可用时：

- **适合并行：** 独立功能切片、已实现功能的测试、文档
- **必须串行：** 数据库迁移、共享状态修改、依赖链
- **需要协调：** 共享 API 契约的功能。先定义契约，再并行实现

## 常见误区

| 误区 | 现实 |
|------|------|
| “边做边想就行” | 这通常会导致纠缠和返工。10 分钟规划能节省数小时。 |
| “任务很明显” | 仍然写下来。显式任务能暴露隐藏依赖和遗漏边界。 |
| “规划是额外开销” | 规划就是任务的一部分。没有计划的实现只是打字。 |
| “我可以全记在脑子里” | 上下文窗口有限。书面计划能跨会话和压缩保留。 |

## 危险信号

- 没有书面任务列表就开始实现
- 任务只写“实现功能”，没有验收标准
- 计划中没有验证步骤
- 所有任务都是 XL 规模
- 任务之间没有检查点
- 未考虑依赖顺序

## 验证

开始实现前确认：

- [ ] 每个任务都有验收标准
- [ ] 每个任务都有验证步骤
- [ ] 任务依赖已识别且排序正确
- [ ] 没有任务触及超过约 5 个文件
- [ ] 主要阶段之间存在检查点
- [ ] 人类已审阅并批准计划
